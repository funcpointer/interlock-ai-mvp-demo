# MVP Audit - 2026-05-25

Scope: every source file, every top-level function, and every major CLI/runtime artifact in the current CLI MVP.

Audit stance: funding-demo MVP, high consequence engineering review. The system should prefer cited, directional, low-noise findings over broad but unverifiable recall. Search, graph, wiki, LanceDB, and memory-palace layers are allowed to improve navigation and recall, but must not publish findings.

## Executive Verdict

The MVP is coherent enough to keep building. The architecture now has the right authority boundary:

- `run_review(ReviewRequest) -> ReviewResult` is the reusable core API.
- Canonical truth is JSON.
- Findings are emitted from reasoning decisions plus cited evidence.
- Search/second-brain/LanceDB/wiki/Kuzu/context memory are derived.
- Near-real fixture behavior is stable.

The main accuracy risk is not storage, graph, search, or memory. It is extraction and context segmentation:

- PyMuPDF text blocks are a good deterministic floor but weak for scans, tables, curves, rotated labels, and image-only content.
- Regex evidence mining is enough for the current planted transformer/fuse/fault-current MVP, not enough for broader AES review packets.
- Context rooms mirror whatever segmentation quality exists upstream.

## Patch Made During Audit

Issue found: scanned/low-text runs produced coverage-warning trails but no meaningful context rooms. First patch collapsed all coverage warnings into one document room per doc because coverage evidence inherited `A:document` / `B:document`.

Fix:

- `ContextMemory` now creates page-specific coverage rooms: `A:coverage_p1`, `B:coverage_p9`, etc.
- Scanned run now has 18 rooms, 18 trails, and 18 rooms with findings.
- Added regression test: coverage warnings create page-level rooms and trail links.

This is the right MVP behavior. Scanned pages are not extractable, but the reviewer still gets a navigable map of failed pages.

## Findings / Issues

### P0

None found. Current MVP contracts pass.

### P1

1. **Extraction is the accuracy ceiling.**
   - Regex + PyMuPDF will not handle the full AES packet reality.
   - Evidence: scanned fixture correctly produces only coverage warnings.
   - Next: add OCR/VLM/table extraction against real AES docs.

2. **Context segmentation is the memory-palace ceiling.**
   - Rooms/trails are only as good as upstream context labels.
   - Current TCC/table-row detection is useful but brittle.
   - Next: improve context extraction with table/page/section structure from real docs.

3. **Candidate generation is auxiliary but still looks first-class.**
   - `candidates.json` exists, but final findings come from reasoning decisions.
   - This is okay if documented. Avoid extending `core/candidates.py` before decision-sourced matching is fully primary.

### P2

4. **Kuzu does not mirror context-memory rooms/trails yet.**
   - Not blocking because JSON/wiki/search already include them.
   - Add only if graph inspection becomes important for demo/debug.

5. **Cloud verifier is a screen, not a real multimodal verifier.**
   - Correct for no-cloud deterministic MVP.
   - For accuracy on scans/rotated text, VLM needs to verify cited crops directly.

6. **Doc-type classification is heuristic.**
   - Use corpus manifest overrides for real AES docs.
   - Do not trust filename/content heuristics alone when authority matters.

7. **CLI has low direct unit coverage.**
   - Core is tested. CLI smoke coverage can be added if packaging becomes a submission risk.

## What Is Right for MVP

Keep:

- CLI-first core API.
- JSON-as-truth.
- Directional authority.
- No finding without citations.
- Deterministic verifier as final gate.
- Eval YAML as acceptance contract.
- Corpus manifest for AES docs.
- Memory/wiki/search as derived navigation.
- Coverage warnings for scans.

Avoid:

- More regex patches as the main accuracy strategy.
- Making LanceDB/search/wiki create findings.
- Hiding scan failure behind low-confidence claims.
- Broad UI work before real AES corpus behavior is known.
- Tuning on synthetic fixtures only.
